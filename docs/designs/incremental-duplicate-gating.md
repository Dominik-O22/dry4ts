# CEO Plan: Incremental Duplicate Gating
Generated by /plan-ceo-review on 2026-06-13
Branch: main | Mode: EXPANSION
Repo: Dominik-O22/dry4ts

## Problem

`--fail-on-duplicates` exits 1 when `clusters.length > 0` (src/DryTs.ts:54). Zero-tolerance: any duplicate anywhere fails the build. No real codebase is ever fully DRY, so the flag is unusable as a gate. The available workaround — raising `--threshold` / `--min-lines` / `--min-nodes` / `--min-locations` — is the wrong lever: those knobs control *detection sensitivity*, and cranking them to silence the gate also blinds the report. Detection and gating policy are conflated.

Primary consumers, equally weighted:
1. **AI agents mid-coding-loop**: after editing, run dry-ts and self-correct if the edit introduced duplication. Must be stateless, fast, machine-readable.
2. **CI**: fail only when a change introduces *new* duplication, tolerate known debt.

## Vision

### 10x Check
dry-ts stops being a detector you run occasionally and becomes a duplication *regression gate*: "no change makes the codebase wetter." Stateless — no checked-in baseline file, no maintenance, no merge conflicts (the classic ESLint/PHPStan-baseline pain). jscpd's line-%-threshold can't express this; dry-ts's structural fingerprints + location spans can.

### Platonic Ideal
An agent edits code, runs `dry-ts --changed src/foo.ts --changed src/bar.ts --format json`, sees exactly the clusters its edit touched (`status: "new"`), extracts the shared helper, re-runs, exits 0, ships. A human's PR pipeline runs `dry-ts --changed-from origin/main --fail-on-duplicates` and goes red only when the PR copied something — with the existing counterpart location named. Five minutes from install to a meaningful gate on a 400-duplicate codebase, zero config files.

## Chosen Approach (D10, supersedes D2)

**Incremental / diff-based gating.** A cluster is a *finding* iff ≥1 of its locations intersects changed code. Effort: human ~2-3 days / CC ~2-3 h including tests and docs (re-estimated in eng review after absorbing 11 review additions; the original ~45-60 min predated them).

### Pipeline (eng review 3A)

```
              ┌─ --changed-from <ref> ─► GitProvider ──► hunks ─┐
  CLI args ───┤                          (merge-base,           ├─► ChangedRegions ─┐
              │                           diff -U0 -M,          │   (file → sorted  │
              └─ --changed <file>... ───► every line of file ───┘    line ranges,   │
                                                                     canonical paths)│
                                                                                     ▼
  scan paths ───► FileScanner ───► Finder ───► clusters ───► intersect ───► status per cluster
                  (unchanged)      (unchanged)               (≥1 location      new | known | unscoped
                                                              overlaps a            │
                                                              changed range)        ▼
                                                                          formatters (json/edn/text)
```

```
  EXIT CODES (total — every termination path defined)
  parse CLI args ── bad flag / both scope flags / bad value ──────────────► exit 2
       │
  resolve changed regions ── git missing / bad ref / not a repo /
       │                     unrecognized diff line / any git failure ────► exit 2
  scan + cluster ── scanner throw (unparseable file, IO error) ───────────► exit 2
       │            zero files scanned under --fail-on-duplicates ────────► exit 2
  intersect + report
       ├── findings (status "new") + --fail-on-duplicates ────────────────► exit 1
       └── otherwise ─────────────────────────────────────────────────────► exit 0
```

- Cluster entirely in unchanged code → known debt, reported but never gates.
- Cluster with one location in changed code + counterpart in old code → "you copied existing code" → finding.
- Cluster entirely inside changed code → new duplication → finding.

Key implementation insight: **no base-ref scan needed.** Semantics is computable from a single current-tree scan: intersect each cluster's location line-ranges with changed ranges. Changed ranges come from exactly two sources (no auto mode in this plan; default without either flag = unscoped scan, fully back-compat):
- `--changed-from <git-ref>` — line-level: changed line-ranges from `git diff -U0 -M <ref>` hunks (zero context — default ±3 context lines would silently widen the gate; **post-image** line ranges). The diff is **ref → working tree**, matching the files the scanner reads, so line numbers align even with a dirty tree. **Untracked-file rule (eng review C2, supersedes the `ls-files --others` design):** a scanned file counts as **fully changed iff it is not in git's index** (`git ls-files`, pathspec-bounded to the scan paths). This is decided per file the scanner actually read, so it is immune by construction to divergence between the scanner's ignore logic (cwd `.gitignore` only) and git's full ignore stack (global excludes, `.git/info/exclude`, nested `.gitignore`) — no `--exclude-standard` / `--no-gitignore` special case needed; without this rule, an agent creating a brand-new duplicate file would bypass the gate. Renames are followed (`-M`), with **only edited hunks counting as changed** in a renamed file (decided; test to this). A file renamed *into* scan scope with no edits gates nothing — intended: moving code is not duplicating it (document in README). **Deletion-only hunks mark the post-image boundary line as changed** (eng review C1, supersedes "pure deletions mark nothing"): deleting the statement that distinguished a function can make the remaining code a duplicate; the `+c,0` hunk marks line `c` so that cluster gates as `new` (fixture test required).
- `--changed <file>` — repeatable flag (`--changed a.ts --changed b.ts`), file-level granularity: every line of the named file counts as changed. Repeatable form avoids variadic ambiguity against positional scan paths in the existing parser (src/Options.ts).

Passing both `--changed-from` and `--changed` is a usage error (exit 2). A changed file that falls outside the scanned paths triggers a warning on stderr (the gate cannot see it). **Canonical path form (eng review C3):** all internal location keys are **repo-root-relative with forward slashes**; git output and scanner paths (OS-native, cwd-relative) are both converted at the module boundary — one form, one conversion site, tests for cwd ≠ repo root and Windows separators. Flag names are **locked** (eng review 8A): `--changed-from` / `--changed`.

Architecture note: gating engine consumes a "changed-region set." A future persisted baseline file is just an alternative provider on the same interface — deferred, not foreclosed. Component boundary: a pure `ChangedRegions` module (hunk parsing → ranges; intersection → status) unit-testable without git, with the git subprocess isolated behind a thin provider — same one-module-per-concern pattern as FileScanner/Normalizer/Finder.

Rejected alternatives:
- Budget flags (`--max-clusters N`): arbitrary, goes stale, lets new dup hide under old headroom (D2, D8).
- Checked-in baseline file as core: stateful, merge conflicts, version invalidation, pre-snapshot step for agents (D10; was briefly chosen in D2, superseded).

## Accepted Scope (this plan)

- `--changed-from <git-ref>`: changed line-ranges from git diff hunks (ref → working tree), union untracked files; flag intersecting clusters. Line-level granularity is **decided**, not open.
- `--changed <file>` (repeatable): explicit changed-file list, file-level granularity, for agents/non-git callers. Combining with `--changed-from` = usage error (exit 2).
- `--fail-on-duplicates` composes: in incremental scope it fails only on findings (clusters with `status: "new"`); unscoped behavior unchanged (back-compat) — note the unscoped combination exits 1 while every cluster reports `status: "known"`; document this explicitly in README/AGENTS.md so agents reading `status` aren't confused.
- Output `status: "new" | "known" | "unscoped"` per cluster in **all formats** (JSON `status`, EDN `:status`, text marking): **always emitted** — one stable output shape, no conditional fields. Contract: `status = "new"` iff the cluster has ≥1 location intersecting the changed set; `"known"` otherwise; `"unscoped"` for every cluster when no changed-scope is active (the tool cannot know what's known without a scope — claiming `"known"` would be a machine-readable lie; cross-model finding, D18). ("new" marks the *finding*, even when the counterpart location is old code.)
- Text output: findings surfaced distinctly (exact formatting in eng/design review).
- **Spawn discipline (security)**: all git invocations use an argument vector (execFile-style, never a shell string) — CI refs are attacker-influenced on fork PRs. Refs starting with `-` are rejected outright (argument-injection class: `--output=...` would be parsed as a git option); the ref is validated with `git rev-parse --verify '<ref>^{commit}'` before use (rejects non-commit objects, not just unresolvable names). Injection tests included.
- **Merge-base resolution (cross-model finding, D19)**: `--changed-from <ref>` does NOT diff literally against `<ref>` — it resolves `git merge-base <ref> HEAD` and diffs from that. A branch behind its base would otherwise see unrelated base-side changes pollute the changed set and produce false gate failures. Users write `--changed-from origin/main` and get correct semantics; no shell-level merge-base incantation in recipes.
- **Fail-closed error rule**: ANY git provider failure — git binary missing (spawn ENOENT), non-zero exit for any reason, or diff output the hunk parser does not recognize — is exit 2 with a dry-ts message that includes git's stderr. **Extended to the scanner (eng review 1A):** any error in the scan/cluster pipeline (unparseable source file, IO error) is caught in `main()` and exits 2 with a message — never an uncaught throw, which would exit 1 and read as "findings" to CI/agents. Exit codes are the gate's API; every termination path has a defined meaning (see exit-code diagram). The hunk parser is strict, never best-effort: an unrecognized line is an error, not a skip. To keep strictness from becoming cry-wolf, the parser's grammar explicitly recognizes (and deliberately skips) ALL legitimate non-hunk forms (eng review 6A completed the list): binary-file markers ("Binary files … differ" → skip file), `\ No newline at end of file` markers, rename headers (`similarity index`, `rename from`, `rename to`), submodule commit entries, and mode-change lines — each with its own fixture test. Quoted-path handling is **decided**: git runs with `-c core.quotePath=false` (no unquoting code path). **Git invocation hardening (eng review C3):** every diff/ls-files call also passes `--no-color --no-ext-diff --no-textconv` and separates pathspecs with `--`, so user/global git config (diff drivers, textconv, color) can never reshape output under the strict parser. A gate must fail loudly; a silently wrong changed set would wave new duplication through green. Tests: git-missing, generic git failure, malformed diff output, scanner-error-under-gating.
- **CLI surface hardening (eng review 4A)**: unknown arguments starting with `-` are rejected (exit 2, naming the bad flag) instead of silently becoming scan paths — today `dry-ts --fail-on-duplicates --changed-frm main` would scan nothing and exit 0 (Options.ts:109-110 + nonexistent paths silently skipped). Additionally, zero files scanned under `--fail-on-duplicates` is an error (exit 2), not a green build. Tests for both.
- Exit codes: 0 = success (with `--fail-on-duplicates` + a changed-scope: no new duplication; findings without the fail flag still exit 0), 1 = findings with `--fail-on-duplicates`, 2 = usage error (bad ref, not a git repo with `--changed-from`, both scope flags). Documented in README + AGENTS.md.
- A `--changed` file outside scanned paths or nonexistent: **without** `--fail-on-duplicates`, stderr warning, exit unaffected; **with** `--fail-on-duplicates`, exit 2 — an ungateable changed file under active gating is a configuration error, and a green build with an ignored warning is a silent bypass (cross-model finding, D20). No analogous treatment for `--changed-from` hunks outside scan paths (intentional — would be noisy on broad diffs).
- `--explain-changed`: dump the resolved changed-region map (file → line ranges, with source: hunk / untracked / listed) to stderr, so confusing gate results are self-diagnosable in one rerun — path normalization is the likeliest bug source and this makes it observable.
- Onboarding: README CI recipe (merge-base pattern for PRs, **including the `fetch-depth` note** — default shallow checkouts break merge-base), AGENTS.md agent-loop recipe ("after editing, run with `--changed <file>`; if exit 1, refactor clusters with status new"). **File-granularity caveat (eng review 2B, resolves boy-scout open question):** AGENTS.md states explicitly that `--changed <file>` scopes the *whole file*, including pre-existing duplication in it; the recipe steers git-aware agents to `--changed-from HEAD` for uncommitted edits (`HEAD~1`/merge-base if the agent commits before gating) for line precision. Text output words findings as "intersects your change", never "you created this".
- Public API (eng review C3): `status` is an **additive** field on the exported `Cluster`/`ClusterReport` types (src/types.ts) — minor version bump per the 0.3.0 precedent; library callers without a changed-scope see `"unscoped"`. `OutputFormat` tightens to the closed union `"text" | "edn" | "json"` (eng review 5A) so format switches are compile-time exhaustive.
- `--changed <path>` that names a directory: same rule as nonexistent files (D20 symmetry) — exit 2 under `--fail-on-duplicates`, stderr warning otherwise. Test to this.
- Tests: git integration tests use a **hermetic fixture helper** (explicit `-c user.name`/`-c user.email`/`-c commit.gpgsign=false` or `GIT_CONFIG_GLOBAL=/dev/null`, explicit initial branch) so runner-global git config can never flake the suite. Full coverage of intersection semantics — line-level, file-level, boundary lines, deletion-only hunks (boundary line must flag when remaining code became duplicate — C1 fixture), untracked new files (must flag, index-based rule incl. an ignore-stack-divergence case), renamed files with edits (only edited hunks flag), **stale-branch merge-base fixture** (branch behind base: base-side changes must NOT pollute the changed set — pins D19), path canonicalization (cwd ≠ repo root, relative scan paths, separator form), changed-file-outside-scan-paths and nonexistent/directory `--changed` args, both-flags usage error, non-git dir errors, empty diff, `--explain-changed` output assertions, **per-format `status` assertions** (json/edn/text incl. `"unscoped"`), unknown-flag rejection, zero-files-under-gate, scanner-error exit codes. Regression guard: unscoped `--fail-on-duplicates` behavior pinned by existing test/dry-ts.test.ts:1090 — must stay green.
